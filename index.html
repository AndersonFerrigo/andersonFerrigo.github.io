fase_00
Fundação silenciosa — o que a pós não ensina
Antes de padrões arquiteturais e pipelines, existem três pilares invisíveis que fazem toda a diferença. Sem eles, você vai decorar conceitos. Com eles, vai entender mecanismos.
⏱ 2 meses · Nível: iniciante → intermediário · Extra além da pós
⚡ por que isso vem antes de tudo

Imagina tentar consertar um motor sem entender como a combustão funciona. Você pode apertar parafusos — mas quando algo der errado, você não vai saber o porquê. SOLID, Design Patterns e networking são o combustível que faz tudo fazer sentido. Arquitetos que não têm essa base sólida tomam decisões por convenção, não por raciocínio.

// tópicos aprofundados
🧱
SOLID — os 5 princípios que explicam o Spring Boot
Você provavelmente já aplica SOLID sem saber o nome. O Spring usa cada princípio internamente:

S — Single Responsibility: cada bean faz uma coisa só (@Service, @Repository).
O — Open/Closed: você estende comportamento com @Primary e interfaces, não reescreve código.
L — Liskov Substitution: seus repositórios JPA implementam interfaces — substitutos transparentes.
I — Interface Segregation: JpaRepository vs CrudRepository — interfaces específicas para cada caso.
D — Dependency Inversion: o coração da injeção de dependência do Spring.

Meta: conseguir identificar e nomear violações de SOLID em um code review.
ArquiteturaExtra
🎨
Design Patterns (GoF) — o vocabulário do arquiteto
Não precisa decorar todos os 23 padrões. Precisa entender os mais usados no ecossistema Java:

Factory / Abstract Factory: como o Spring cria beans.
Singleton: escopo padrão de todo bean Spring.
Strategy: troca de algoritmos em tempo de execução — base de qualquer regra de negócio extensível.
Observer: base dos eventos Spring (ApplicationEventPublisher) e do Kafka.
Decorator: como o Spring AOP funciona por baixo dos panos.
Builder: você já usa no Lombok com @Builder.
CQRS pattern: separação Command/Query que vai aparecer em 3 fases deste roadmap.

Meta: quando alguém falar "usa Strategy aqui", você saber implementar imediatamente.
ArquiteturaExtra
🌐
Networking & protocolos — a anatomia do arquiteto
Um arquiteto de microsserviços que não entende networking é como médico sem anatomia. Você não precisa ser sysadmin — precisa entender o suficiente para tomar decisões:

HTTP/1.1 vs HTTP/2 vs HTTP/3: por que HTTP/2 multiplexado muda a equação de microsserviços.
TCP vs UDP: por que Kafka usa TCP e não UDP.
DNS & Load Balancing: o que acontece antes do seu Spring Boot receber a requisição.
TLS/mTLS: como microsserviços se autenticam mutuamente — crítico para DevSecOps.
REST vs gRPC vs GraphQL: cada um tem trade-offs reais — aprenda quando escolher cada um.

Meta: conseguir explicar o que acontece nos bastidores de uma chamada HTTP entre dois microsserviços.

📐
Diagramas C4 — a linguagem visual do arquiteto
O modelo C4 (Simon Brown) é o padrão de facto para documentar arquitetura no mercado brasileiro. São 4 níveis:

Context: seu sistema no mundo — o que ele faz e com quem interage.
Container: as "caixas" que compõem seu sistema (API, banco, fila, frontend).
Component: o que tem dentro de cada caixa (controllers, services, repos).
Code: diagramas UML se necessário (geralmente desnecessário).

Ferramentas: use Excalidraw (grátis, colaborativo) ou Structurizr (C4 nativo, tem free tier).

Meta: documentar sua API atual com C4 Nível 1 e 2 antes de terminar este mês.

Padrões Arquiteturais — do diagrama à decisão
A pós ensina os padrões. Esta fase ensina a raciocinar sobre eles: quando usar, quando não usar, como defender uma escolha para um time e para um CTO.
⏱ 2 meses · Nível: intermediário · Alinha com disciplina da pós: Padrões Arquiteturais
⚡ a virada de mentalidade

Devs escolhem tecnologias. Arquitetos escolhem trade-offs. A pergunta nunca é "qual padrão é melhor?" — é "quais são os requisitos não-funcionais deste sistema? Qual padrão resolve o maior problema com o menor custo?". Uma pesquisa de 2025 analisando 411 vagas de arquiteto no Brasil mostrou que a skill mais cobrada não é uma tecnologia, é a capacidade de "planejar e implementar atividades arquiteturais".

// tópicos aprofundados
🔀
Monólito vs Microsserviços — a decisão honesta que ninguém ensina
95% dos sistemas que "precisam" de microsserviços não precisam. Entender isso é o que separa um arquiteto honesto de um que segue tendências.

Microsserviços fazem sentido quando: times independentes precisam fazer deploy sem coordenar entre si, domínios de negócio têm ciclos de vida e requisitos de escala muito distintos, e você tem maturidade de operação para gerenciar a complexidade.

Monólito modular faz sentido quando: time pequeno/médio, domínio não está completamente mapeado ainda, necessidade de velocidade de desenvolvimento.

Padrão Strangler Fig: como migrar um monólito gradualmente sem reescrever tudo — cria uma "casca" ao redor do sistema legado e vai substituindo partes incrementalmente. Sua API atual provavelmente é candidata a esse padrão.
Arquitetura
📡
Event-Driven Architecture (EDA) — desacoplamento real
Em vez do Serviço A chamar o Serviço B diretamente (acoplamento temporal e de disponibilidade), A publica um evento numa fila. B consome quando puder. Se B cair, o evento ainda está lá esperando.

Conceitos essenciais:
Event — algo que aconteceu: "CicloDeAvaliação.Finalizado"
Command — uma intenção: "FinalizarCicloDeAvaliacao"
Eventual Consistency — o estado do sistema converge ao longo do tempo, não é imediato
Outbox Pattern — garante que o evento seja publicado mesmo se o banco e o Kafka transacionarem separados

Para o seu contexto: todo ciclo de avaliação finalizado poderia ser um evento que dispara múltiplos consumidores: notificação, relatório analítico, log de auditoria.
ArquiteturaExtra
✂️
CQRS + Event Sourcing — leitura e escrita separadas
CQRS (Command Query Responsibility Segregation): o modelo de escrita e o modelo de leitura são diferentes. Você escreve em um banco normalizado, e lê de uma view otimizada (pode ser Redis, Elasticsearch, ou outra tabela).

Por que isso importa para dados: quando você separar leitura de escrita, a "leitura" pode ser um pipeline analítico — você acabou de criar a ponte entre arquitetura transacional e arquitetura de dados.

Event Sourcing: em vez de salvar o estado atual, você salva todos os eventos que geraram esse estado. O histórico completo de mudanças vira o dado em si — base natural para auditoria, replay e análise temporal. Pense em seu sistema de avaliações: cada mudança de nota, cada aprovação, cada feedback vira um evento imutável.
ArquiteturaDados
🗺️
Domain-Driven Design (DDD) — o mapa que faltava
DDD é a forma de modelar software com base no vocabulário do negócio, não da tecnologia. É o conceito mais impactante que a pós não cobre diretamente.

Bounded Context: uma fronteira clara onde um conceito tem um significado específico. "Funcionário" no contexto de RH é diferente de "Funcionário" no contexto de Folha de Pagamento — são entidades distintas com atributos distintos.

Aggregate: um cluster de objetos com uma raiz consistente — só você acessa de fora. CicloDeAvaliacao com seus Itens provavelmente é um Aggregate.

Ubiquitous Language: o vocabulário que devs e especialistas de negócio compartilham. Quando um produto fala "ciclo" e o código chama "period", existe uma lacuna — DDD elimina isso.

Para microsserviços: cada Bounded Context é um candidato natural a microsserviço independente. Este é o critério real de separação, não "separar por tamanho de código".
Arquitetura
📊
Architecture Decision Records (ADRs) — como arquitetos registram decisões
Um ADR é um documento curto que registra uma decisão arquitetural: o contexto, as opções consideradas, a decisão tomada e as consequências. É a prova documental de que você pensa como arquiteto.

Template básico: Título, Status (proposto/aceito/substituído), Contexto, Decisão, Consequências.

Por que isso importa nas entrevistas: arquitetos são cobrados por suas decisões passadas. Ter ADRs no seu portfólio GitHub mostra maturidade de processo — não é só quem fez, é quem documentou o raciocínio.

Ferramenta: adr.github.io tem templates prontos e ferramentas CLI para gerenciar ADRs no repositório.

Containers, Kafka e Microsserviços na prática
A pós fala sobre microsserviços. Esta fase é onde você coloca a mão no terminal. Não existe arquiteto de microsserviços que nunca rodou um container. Esta é a ponte entre o diagrama e a realidade.
⏱ 2 meses · Nível: intermediário · Extra além da pós — mas pré-requisito do mercado
⚡ por que está aqui e não na pós

Docker e Kafka não estão na ementa, mas aparecem como requisito em vagas de arquiteto no Brasil. Isso não é coincidência — são as ferramentas que transformam diagramas em sistemas reais. Um arquiteto que nunca containerizou uma aplicação não consegue validar suas próprias decisões.

// tópicos aprofundados
🐳
Docker — containerize seu Spring Boot em menos de 1 dia
Um container é uma "caixa portátil" com tudo que sua aplicação precisa: JDK, dependências, variáveis de ambiente. Elimina o "funciona na minha máquina".

O que você precisa saber:
Dockerfile — como sua imagem é construída (FROM, COPY, RUN, EXPOSE, ENTRYPOINT)
docker-compose.yml — como múltiplos serviços orquestram juntos (API + PostgreSQL + Kafka)
Multi-stage builds — como criar imagens menores e mais seguras (build separado do runtime)
Volumes e networks — como dados persistem e serviços se comunicam

Primeiro exercício: crie um Dockerfile para sua API Spring Boot. Suba com docker-compose junto com o PostgreSQL. Quando funcionar, você entendeu Docker.
ExtraGRÁTIS
⚡
Apache Kafka — o sistema nervoso de microsserviços
Kafka é um sistema de streaming distribuído. Em vez de A chamar B diretamente, A publica uma mensagem num topic. B consome quando disponível. O dado fica armazenado por tempo configurável.

Conceitos em ordem de aprendizado:
Topic — um "canal" de mensagens (ex: ciclos-finalizados)
Producer — quem publica mensagens (sua API Spring Boot)
Consumer — quem consome (relatórios, notificações, pipeline de dados)
Partition — divisão paralela de um topic (escalabilidade horizontal)
Consumer Group — múltiplas instâncias de um consumer dividindo o trabalho
Offset — a posição de leitura de cada consumer — Kafka nunca "apaga" por padrão

Para Java devs: Kafka usa Java nativamente. O spring-kafka transforma producers e consumers em simples anotações: @KafkaListener.

☸️
Kubernetes (K8s) — quando docker-compose não é suficiente
Se Docker é "empacotar", Kubernetes é "gerenciar muitos containers em produção". Cuida de restart automático, scaling horizontal, health checks e rollouts sem downtime.

Os 4 objetos que você precisa dominar primeiro:
Pod — a menor unidade: um ou mais containers rodando juntos
Deployment — declara quantas réplicas do seu Pod você quer e gerencia rollouts
Service — expõe seu Pod com um IP estável (load balance interno)
ConfigMap / Secret — variáveis de ambiente e credenciais desacopladas da imagem

Estratégia de estudo: comece com minikube local ou Kind (Kubernetes in Docker) — sem custo de cloud. Suba sua API Spring Boot no K8s local antes de ir para AWS.
Extra
🔌
API Gateway + Service Mesh — o que vem depois dos microsserviços
Quando você tem 5+ microsserviços, o caos de comunicação precisa de gestão centralizada:

API Gateway (Kong, AWS API Gateway, Spring Cloud Gateway): ponto único de entrada para clientes externos. Cuida de autenticação, rate limiting, roteamento e logging.

Service Mesh (Istio, Linkerd): gestão de comunicação entre serviços internos — mTLS automático, circuit breakers, observability. É a "rede inteligente" invisível.

Circuit Breaker (Resilience4j no Spring): quando um serviço está falhando, você "abre o circuito" para evitar cascata de falhas. Conceito crítico para arquiteto — aprenda Resilience4j que já integra com Spring Boot.

Ciência de Dados — do pipeline ao insight
A pós cobre Agile Data Science e Big Data em alto nível. Esta fase é onde você entende o que acontece dentro de um pipeline real e como arquitetar sistemas que servem ciência de dados — com as ferramentas que o mercado paga mais.
⏱ 2 meses · Nível: intermediário → avançado · Alinha com disciplinas da pós: Ciência de Dados + Big Data
⚡ a conexão que poucos fazem

Engenheiros que entendem de dados valem mais porque são raros. Você já sabe construir APIs que geram dados — agora aprenda a arquitetar sistemas que transformam esses dados em inteligência de negócio. Kafka (fase anterior) é a entrada dos dados. // tópicos aprofundados
🗃️
SQL avançado — a base que você já tem, mas não aprofundou
PostgreSQL já é sua especialidade. Mas SQL analítico é diferente de SQL transacional. Antes de ir para NoSQL e Spark, aprofunde o que você já tem:

Window Functions — ROW_NUMBER(), RANK(), LAG(), LEAD(), SUM() OVER(PARTITION BY). Essenciais para relatórios analíticos sem precisar de Spark.

CTEs (Common Table Expressions) — consultas legíveis e reutilizáveis dentro de um SELECT. Substitui subqueries aninhadas.

Query Planning — EXPLAIN ANALYZE: entender como o PostgreSQL executa sua query e onde está o gargalo. Crítico para otimização de APIs com queries lentas.

Particionamento de tabelas — divide uma tabela grande em partes por data, ID ou outra coluna. Reduz drásticamente o tempo de queries analíticas em tabelas com milhões de linhas.
Dados
🔄
ETL/ELT — o coração de qualquer pipeline de dados
ETL (Extract, Transform, Load): extrai de uma fonte, transforma os dados, carrega no destino. A transformação acontece antes de chegar no warehouse.

ELT (Extract, Load, Transform): carrega bruto primeiro, transforma dentro do warehouse (BigQuery, Snowflake). Mais comum em arquiteturas modernas porque os warehouses são poderosos o suficiente para transformar.

Tipos de carga:
Full Load — recarrega tudo. Simples, mas caro em dados grandes.
Incremental Load — só o que mudou desde a última carga. Precisa de watermark (timestamp ou ID crescente).
CDC (Change Data Capture) — captura mudanças direto do WAL (Write-Ahead Log) do banco. Ferramenta: Debezium + Kafka.

Para o seu contexto: os eventos Kafka da fase anterior já são um pipeline ELT: o consumer extrai (consome o Kafka), carrega no banco analítico, e a transformação pode acontecer depois com Spark ou dbt.
Dados
⚡
Apache Spark — processamento distribuído em Java
Spark é escrito em Scala/Java. Se você já tem fluência em Java, a API do Spark vai parecer familiar. É o motor padrão para processar dados que não cabem em memória de uma máquina.

Sequência de aprendizado:
1. RDD (Resilient Distributed Dataset) — a abstração base. Entenda o conceito mesmo sem usar muito no dia a dia.
2. DataFrame API — similar ao SQL/Pandas. Você vai passar 90% do tempo aqui.
3. Spark SQL — escreva SQL direto no Spark contra DataFrames. Ponte natural para quem já sabe SQL.
4. Spark Structured Streaming — processa Kafka em tempo real com a mesma API de batch. É aqui que os eventos da fase 2 se transformam em dados analíticos em tempo quase real.

Otimizações essenciais: particionamento, caching, broadcast joins, evitar shuffles desnecessários.
DadosExtra
🌊
Apache Airflow — o maestro dos pipelines
Airflow é o orquestrador de workflows de dados mais usado no mundo. Você descreve tarefas como um DAG (Directed Acyclic Graph) em Python — uma série de passos com dependências.

Exemplo de DAG para o seu sistema:
extrai_dados_kafka → processa_com_spark → carrega_no_warehouse → atualiza_dashboard

Por que não apenas cron jobs? Airflow tem retry automático, alertas de falha, visualização gráfica do pipeline, e histórico de execuções. É a diferença entre pipeline artesanal e pipeline de produção.

Alternativas modernas: Prefect e Dagster têm APIs mais amigáveis para Python, mas Airflow ainda domina o mercado corporativo brasileiro.
DadosExtra
🏛️
Data Warehouse vs Data Lake vs Data Lakehouse — o triângulo da decisão
Como arquiteto, você vai ser perguntado sobre isso em entrevistas. Aprenda os critérios reais de decisão:

Data Warehouse (Snowflake, BigQuery, Redshift): dados estruturados, schema definido, otimizado para queries analíticas. Caro, mas muito rápido para BI. Use quando os dados já chegam limpos.

Data Lake (S3 + Parquet, HDFS): dados brutos em qualquer formato. Barato, flexível, mas precisa de governança forte para não virar um "data swamp".

Data Lakehouse (Delta Lake, Apache Iceberg, Apache Hudi): combina o melhor dos dois — armazenamento barato do lake com capacidade transacional e ACID do warehouse. É o padrão mais moderno e o que você vai construir no projeto prático.

NoSQL para dados operacionais: MongoDB (documentos flexíveis), Redis (cache + sessão + filas), Cassandra (writes massivos com alta disponibilidade — usado pelo Itaú e grandes bancos).

AWS para arquitetos — os serviços que importam
Não precisa saber tudo da AWS. Precisa saber o que cada serviço resolve e quando escolher cada um:

Compute: EC2 (servidor virtual — controle total), ECS/Fargate (containers gerenciados), Lambda (serverless — pay per use), EKS (Kubernetes gerenciado)

Armazenamento: S3 (objetos — data lake, backups, static assets), EBS (disco para EC2), EFS (filesystem compartilhado)

Banco de dados: RDS (PostgreSQL gerenciado — zero ops), DynamoDB (NoSQL serverless), ElastiCache (Redis gerenciado)

Mensageria para dados: MSK (Kafka gerenciado), SQS (filas simples), Kinesis (streaming de dados similar ao Kafka)

Dados: Glue (ETL serverless), Athena (query SQL no S3), Redshift (data warehouse), EMR (Spark gerenciado)

CI/CD com GitHub Actions — do commit ao deploy automático
Pipeline de entrega contínua: código commitado → testes rodam → build Docker → push para ECR → deploy no ECS/K8s. Sem intervenção manual.

Para Java/Spring Boot, o pipeline padrão é:
checkout → mvn test → mvn package → docker build → docker push → kubectl apply (ou ECS deploy)

Conceitos de qualidade no pipeline: SAST (análise estática de segurança — SonarCloud grátis para open source), cobertura de testes mínima, e quality gates que bloqueiam o deploy se algo falhar.

Branch strategy: aprenda GitFlow vs trunk-based development — arquitetos precisam opinar sobre estratégia de branches, não só código.
Extra
🏗️
Terraform — Infrastructure as Code básico mas sólido
Em vez de criar recursos na AWS clicando no console (que não tem versionamento, não é reproduzível, e quebra quando alguém esquece o que fez), você descreve a infraestrutura em código HCL.

O que você precisa dominar para começar:
provider — qual cloud (AWS, GCP, Azure)
resource — o que criar (EC2, RDS, S3, VPC)
variable — parametrização (ambiente, tamanho da instância)
output — o que retornar após apply (endereço do banco, ARN do bucket)
state — onde o Terraform guarda o estado atual da infra

Meta: criar com Terraform o ambiente AWS para o projeto prático (VPC, RDS, S3, ECS ou EC2 para a API).
Extra
📈
Observabilidade — métricas, logs e traces distribuídos
Em sistemas distribuídos, "o servidor caiu" não é suficiente. Você precisa saber qual serviço falhou, em qual instância, qual request causou isso, e quanto tempo cada operação levou.

Os três pilares:
Métricas — números ao longo do tempo: requests/s, latência p99, CPU. Stack: Prometheus + Grafana
Logs — eventos textuais estruturados (JSON). Stack: ELK (Elasticsearch + Logstash + Kibana) ou CloudWatch
Traces — o rastro de uma requisição através de múltiplos serviços. Stack: Jaeger ou OpenTelemetry

Para Spring Boot: Micrometer (incluído no Actuator) exporta métricas automaticamente para Prometheus. OpenTelemetry tem SDK Java que instrumenta automaticamente.

Data Mesh — dados como produto de domínio
Data Mesh é o paradigma mais moderno de arquitetura de dados: em vez de um data lake centralizado (que vira um "data swamp" gerenciado por um time sobrecarregado), cada domínio de negócio é dono dos seus dados e os expõe como produto.

Os 4 princípios do Data Mesh:
1. Domain Ownership: o time de RH é dono dos dados de RH — não o time central de dados
2. Data as a Product: cada domínio expõe seus dados com qualidade, documentação e SLA garantidos
3. Self-serve Data Platform: plataforma que permite times criarem pipelines sem depender do time central
4. Federated Computational Governance: regras globais (LGPD, segurança) aplicadas federadamente

Conexão com DDD: os Bounded Contexts da fase 1 viram Data Products. É o mesmo raciocínio de separação de domínios, agora aplicado aos dados.

MLOps — arquitetando sistemas de Machine Learning
Como arquiteto, você não precisa criar modelos de ML — precisa saber projetar o sistema que os sustenta em produção. Treinar um modelo é 10% do trabalho. Os outros 90% são operação.

Os componentes de um sistema MLOps:
Feature Store — repositório centralizado de features reutilizáveis entre modelos
Model Registry — versionamento de modelos (MLflow)
CI/CD para modelos — pipeline de retreino automático quando dados mudam
Model Monitoring — detecção de data drift (quando o mundo mudou e o modelo ficou defasado)
A/B Testing — comparação de modelos em produção com divisão de tráfego

Ferramenta básica: MLflow (open source) para tracking de experimentos e model registry. Integra com Spark e qualquer linguagem.

Governança de dados — LGPD, lineage e qualidade
Dados sem governança viram problema jurídico e de confiança. A pós cobre LGPD em alto nível — vá além:

Data Lineage: o rastro de onde cada dado veio e como foi transformado. Quando um relatório financeiro estiver errado, você consegue rastrear qual etapa do pipeline corrompeu o dado. Ferramentas: Apache Atlas, OpenLineage.

Data Quality: expectativas formais sobre os dados — "esta coluna não pode ser nula", "este valor deve estar entre 0 e 100". Ferramenta: Great Expectations (Python), mas o conceito é linguagem-agnóstico.

Data Catalog: onde os dados estão, o que significam, quem é dono. Reduz o tempo que engenheiros gastam procurando dados. Ferramentas: Apache Atlas, DataHub (LinkedIn, open source), AWS Glue Data Catalog.
